[设计]抽象工厂模式

场景

  • 其实也是解决工厂模式的不足。
  • 但是不论怎么改工厂的性质是不会变的。所以雷达产品与功能经常性的变化，建议不要采用这样模式。*

    解决办法

  1. 添加抽象工厂的抽象接口是不是就可以添加一种产品了
  2. 添加不同的抽象工厂是不是把实现又向后推了一下，更活性更好了。
  • 我们先创建工厂基本和产品基类
    public abstract  class FactoryBase
  {
      public abstract LidarBase CreatLidar();
      public abstract AirPlaneBase CreatAirPlane();
  }
public abstract class LidarBase
  {
      public abstract void CreateLidar();
  }
public abstract class AirPlaneBase
  {
      public abstract void CreatePlane();
  }
  • 我们再创建具体的产品子类

  • 我们可以创建不同的工厂类了

    class TempFactoryBJ : FactoryBase
  {
      public override AirPlaneBase CreatAirPlane()
      {
          return new TempPlaneBJ();
      }

      public override LidarBase CreatLidar()
      {
          return new TempLidarBJ();
      }
  }
class TempFactorySH : FactoryBase

    注意

  1. 子类过多是不是很多呢，每个新品的变化都会带来很多类的添加
  2. 优点还是那些就是耦合度的降底维护成本底，其实我们可以用反射来进行消除那些switch case的操作,反射影响性能，那只是你用的不好，不要害工具。你轮不起金空棒还怪金空棒沉。
    3.如果一个系统只用产品不在意细节，多个产品只消费其中一个，还有类库的编写，对外提供统一的接口时最好用工厂模式。
